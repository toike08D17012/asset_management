## 3. Detailed Requirements

### REQ-資産管理WEBアプリ-001

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL provide a web interface to aggregate holdings data from multiple securities accounts (stocks and mutual funds) and display them in a unified view

*Rationale*: Core purpose - eliminate manual work of logging into multiple brokerage websites and manually transcribing holdings data to Google Spreadsheet

---

### REQ-資産管理WEBアプリ-002

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL fetch and store stock and mutual fund holdings data including: ticker/fund ID, quantity of shares held, average purchase price per share, and current market price

*Rationale*: These are the essential data points currently being manually transcribed monthly

---

### REQ-資産管理WEBアプリ-003

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL display current holdings, share quantities, and average purchase prices for all stocks and mutual funds across all connected securities accounts

*Rationale*: Success state - eliminate the need to manually compile this data from multiple sources

---

### REQ-資産管理WEBアプリ-004

**Pattern**: event-driven  
**Priority**: P1

> WHEN the user requests data refresh, the system SHALL complete the update process within 3 minutes for up to 2 securities accounts (Rakuten and SBI)

*Rationale*: Performance guideline for development. No user warning required if exceeded. Ensures system remains practical for monthly updates

---

### REQ-資産管理WEBアプリ-005

**Pattern**: unwanted  
**Priority**: P0

> The system SHALL NOT require more than 5 manual user actions (clicks, form inputs, button presses) from accessing the application URL to viewing current aggregated holdings data, IF it does THEN the system SHALL display a warning that manual effort exceeds baseline

*Rationale*: Constraint - system must not increase manual work burden compared to current process. User actions include: URL access, button clicks, form submissions, but exclude page load wait time

---

### REQ-資産管理WEBアプリ-006

**Pattern**: optional  
**Priority**: P2

> IF the user chooses automated sync, THEN the system SHALL fetch holdings data automatically on application startup or user-specified schedule (daily/weekly/monthly)

*Rationale*: Future enhancement. Phase 1: trigger on app startup. Phase 2+: background scheduling (cron/worker)

---

### REQ-資産管理WEBアプリ-007

**Pattern**: ubiquitous  
**Priority**: P2

> The system SHALL be built using modern web development frameworks and tools suitable for learning purposes

*Rationale*: Additional requirement - use project as learning opportunity for modern web development (user has Python background, less web experience)

---

### REQ-資産管理WEBアプリ-008

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL store securities account credentials (login ID, password) in encrypted format using industry-standard encryption algorithms (AES-256 or equivalent)

*Rationale*: Security requirement - protect sensitive financial account credentials from unauthorized access

---

### REQ-資産管理WEBアプリ-009

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL support data fetching from Rakuten Securities (楽天証券) and SBI Securities (SBI証券)

*Rationale*: Target securities firms - these are the two firms currently used by the user

---

### REQ-資産管理WEBアプリ-010

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL provide functionality to add, edit, and delete securities account configurations including account name, securities firm selection, and login credentials

*Rationale*: Account management - users need to configure which securities accounts to track

---

### REQ-資産管理WEBアプリ-011

**Pattern**: event-driven  
**Priority**: P1

> WHEN data fetching from a securities account fails, the system SHALL display an error message indicating which account failed and the reason, while continuing to process remaining accounts

*Rationale*: Error handling - user needs to understand what went wrong and which accounts are affected. Partial success is acceptable and preferred over complete failure

---

### REQ-資産管理WEBアプリ-012

**Pattern**: state-driven  
**Priority**: P1

> WHILE data is being fetched from securities accounts, the system SHALL display a loading indicator showing progress or current status

*Rationale*: UX requirement - user needs feedback during potentially long data fetch operations (up to 5 minutes)

---

### REQ-資産管理WEBアプリ-013

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL store historical snapshots of holdings data including date, ticker/fund ID, quantity, average purchase price, and market price for each data refresh

*Rationale*: Historical tracking - enables trend analysis and asset growth visualization over time

---

### REQ-資産管理WEBアプリ-014

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL display asset value trend graphs showing total portfolio value over time (monthly basis minimum)

*Rationale*: Core feature - user wants to visualize asset growth from monthly dollar-cost averaging investments

---

### REQ-資産管理WEBアプリ-015

**Pattern**: ubiquitous  
**Priority**: P1

> The system SHALL calculate and display total portfolio value, unrealized gain/loss amount, and gain/loss percentage based on average purchase price vs current market price

*Rationale*: Investment analysis - essential metrics for evaluating portfolio performance

---

### REQ-資産管理WEBアプリ-016

**Pattern**: ubiquitous  
**Priority**: P1

> The system SHALL display the last update timestamp for each securities account and for the overall portfolio

*Rationale*: Data freshness - user needs to know how current the displayed data is

---

### REQ-資産管理WEBアプリ-017

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL support manual CSV/HTML file import from securities firms for holdings data. Future phases MAY add semi-automated web scraping with user-assisted 2FA/passkey authentication

*Rationale*: Phase 1 implementation strategy - manual CSV import (楽天/SBI provide downloadable CSV/HTML). Phase 2+: semi-automated scraping requiring user interaction for 2FA. Full automation blocked by passkey/2FA requirements

---

### REQ-資産管理WEBアプリ-018

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL aggregate holdings of the same security across multiple securities accounts and display them as a single consolidated row in the portfolio view

*Rationale*: Data consolidation - when the same stock/fund is held in multiple accounts (e.g., Toyota stock in both Rakuten and SBI), show combined total

---

### REQ-資産管理WEBアプリ-019

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL calculate average purchase price for consolidated holdings using weighted average formula: (qty1 × price1 + qty2 × price2) / (qty1 + qty2)

*Rationale*: Accurate cost basis calculation - when same security held in multiple accounts, weighted average provides correct average purchase price

---

### REQ-資産管理WEBアプリ-020

**Pattern**: event-driven  
**Priority**: P1

> WHEN partial data fetch succeeds (some accounts succeed, others fail), the system SHALL update successfully fetched data and display most recent available data for failed accounts with an indicator showing data staleness

*Rationale*: Resilience - don't block entire update if one account fails. Show best available data with appropriate warnings

---

### REQ-資産管理WEBアプリ-021

**Pattern**: event-driven  
**Priority**: P1

> WHEN no securities accounts are registered on initial application access, the system SHALL display an onboarding screen prompting the user to register their first securities account

*Rationale*: User experience - guide new users through initial setup rather than showing empty state

---

### REQ-資産管理WEBアプリ-022

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL support multiple currencies including Japanese Yen (JPY) and US Dollar (USD) for holdings display and calculations

*Rationale*: Multi-currency support - user holds both domestic Japanese assets and US ETFs/stocks

---

### REQ-資産管理WEBアプリ-023

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL display Japanese domestic assets in JPY and US assets (ETFs, stocks) in both USD and JPY equivalent, showing exchange rate used for conversion

*Rationale*: Currency display - provide both original currency and JPY equivalent for easy comparison. Example: "$10,000 (¥1,500,000 @ ¥150/$)"

---

### REQ-資産管理WEBアプリ-024

**Pattern**: ubiquitous  
**Priority**: P1

> The system SHALL distinguish mutual funds (投資信託) from stocks (株式) in data display, using appropriate icons, labels, or visual indicators

*Rationale*: Asset type clarity - mutual funds and stocks have different characteristics and should be visually distinguishable

---

### REQ-資産管理WEBアプリ-025

**Pattern**: ubiquitous  
**Priority**: P1

> The system SHALL handle mutual fund unit counts (口数) and Net Asset Value (基準価額) appropriately, recognizing that mutual fund prices update once daily (typically next business day) unlike real-time stock prices

*Rationale*: Mutual fund specifics - different pricing model from stocks. NAV updates daily vs real-time stock prices

---

### REQ-資産管理WEBアプリ-026

**Pattern**: ubiquitous  
**Priority**: P1

> The system SHALL store and display dividend and distribution payment history including date, amount, currency, and associated security. SHALL support bulk import from CSV for historical data and automatically track current month's dividends from holdings data refresh

*Rationale*: Income tracking - Primary focus on current month dividends. Historical data (pre-initialization) can be imported via CSV if available

---

### REQ-資産管理WEBアプリ-027

**Pattern**: ubiquitous  
**Priority**: P1

> The system SHALL display dividend and distribution trend graphs showing total income over time (monthly/yearly aggregation)

*Rationale*: Income visualization - see passive income growth from dividend-paying stocks and distribution-paying funds

---

### REQ-資産管理WEBアプリ-028

**Pattern**: ubiquitous  
**Priority**: P1

> The system SHALL calculate and display total return including both capital gains (unrealized gain/loss) and cumulative dividends/distributions received

*Rationale*: Complete performance picture - price appreciation alone doesn't show full investment performance. Total return = capital gains + income received

---

### REQ-資産管理WEBアプリ-029

**Pattern**: ubiquitous  
**Priority**: P2

> The system SHALL provide CSV export functionality for holdings data and historical snapshots

*Rationale*: Data portability - allow user to analyze data in external tools (Excel, Python, etc.) if needed

---

### REQ-資産管理WEBアプリ-030

**Pattern**: ubiquitous  
**Priority**: P1

> The system SHALL generate monthly reports summarizing asset value changes, dividend/distribution income received, and portfolio performance for the month

*Rationale*: Regular reporting - monthly summary helps track progress aligned with monthly dollar-cost averaging investment strategy

---

### REQ-資産管理WEBアプリ-031

**Pattern**: unwanted  
**Priority**: P0

> The system SHALL NOT hardcode encryption keys in source code and SHALL retrieve encryption keys from environment variables or secure configuration files

*Rationale*: Security - encryption keys in source code are a critical security vulnerability. Keys must be externalized

---

### REQ-資産管理WEBアプリ-032

**Pattern**: ubiquitous  
**Priority**: P0

> The system SHALL generate a unique encryption master key on first startup using cryptographically secure random number generator (minimum 256-bit entropy) and SHALL prompt the user to set a master password for key decryption

*Rationale*: Secure key management - Master key encrypts stored credentials. User password protects the master key. Key derivation using PBKDF2 or Argon2

---

### REQ-資産管理WEBアプリ-033

**Pattern**: ubiquitous  
**Priority**: P1

> The system SHALL fetch daily exchange rates (USD/JPY) from external API (Yahoo Finance, Open Exchange Rates, or equivalent free service) and SHALL cache the most recent rate for offline use with timestamp display

*Rationale*: Currency conversion requirement for USD assets. Daily update sufficient (not real-time). Offline fallback ensures app works without internet

---

### REQ-資産管理WEBアプリ-034

**Pattern**: event-driven  
**Priority**: P1

> WHEN CSV/HTML data import fails due to parsing errors or invalid format, the system SHALL display detailed error message indicating the problematic row/field and SHALL allow the user to retry with corrected file

*Rationale*: Error handling for manual import workflow. User needs actionable feedback to fix CSV format issues. No automatic retry (same file = same error)

---

### REQ-資産管理WEBアプリ-035

**Pattern**: ubiquitous  
**Priority**: P2

> The system SHALL provide automatic local backup functionality to export all data (holdings snapshots, account configs excluding credentials, historical data) to timestamped JSON file on weekly basis or manual trigger

*Rationale*: Data safety - protect against data loss from browser storage corruption or accidental deletion. JSON format for portability

---

### REQ-資産管理WEBアプリ-036

**Pattern**: ubiquitous  
**Priority**: P1

> The system SHALL use modern interactive charting library (Chart.js, Recharts, or equivalent) with zoom, pan, time range selection, and tooltip features for asset value and dividend trend graphs

*Rationale*: Enhanced data visualization - interactive features improve analysis capability. Library choice balances ease of implementation with UX quality

---

### REQ-資産管理WEBアプリ-037

**Pattern**: unwanted  
**Priority**: P1

> The system SHALL NOT store unencrypted sensitive data (credentials, master key, decrypted account info) in browser localStorage, sessionStorage, or IndexedDB

*Rationale*: Browser storage security - all sensitive data must be encrypted at rest. Only encrypted blobs stored locally

---

### REQ-資産管理WEBアプリ-038

**Pattern**: ubiquitous  
**Priority**: P2

> The system SHALL log all data fetch operations, errors, and user actions (excluding sensitive data like passwords) with timestamp to application log file with 30-day retention policy

*Rationale*: Troubleshooting and audit trail - helps debug issues without exposing credentials. Automatic log rotation prevents unbounded growth

---

### REQ-資産管理WEBアプリ-039

**Pattern**: optional  
**Priority**: P2

> IF the user chooses to restore from backup, THEN the system SHALL import previously exported JSON backup file and SHALL merge with existing data OR replace entirely based on user selection

*Rationale*: Data recovery - complement to backup feature. Merge option preserves newer data while restoring lost historical records

---

### REQ-資産管理WEBアプリ-040

**Pattern**: unwanted  
**Priority**: P2

> The system SHALL NOT implement responsive mobile UI for smartphone screens (below 768px width). Desktop/tablet screen sizes (1024px+) are the primary target

*Rationale*: Scope limitation - focus development effort on primary use case (PC-based monthly review). Mobile support deferred to future phases

## 4. Additional Context

### User Background
- purpose: 毎月積立で買っている株が存在するため、上記作業を月1回以上実施する必要がある
- targetUser: pythonであれば大体のものは作れる。WEB系はあまり知らない。AI関連はそこそこ詳しい
- constraints: 個人利用しかしない & 大規模なデータを扱う予定はないので、特になし
- successCriteria: 手動で実施が必要な作業の量
- 私の勉強も兼ねて、モダンなフレームワーク・開発環境で開発を実施したいです

### Security Considerations
- No user authentication required (personal use only)
- Securities account credentials must be encrypted (AES-256)
- HTTPS required for all communications

### Target Securities Firms
- Rakuten Securities (楽天証券)
- SBI Securities (SBI証券)

### Historical Data Requirements
- Store snapshots for asset trend visualization
- Monthly granularity minimum
- Support asset growth graphs for dollar-cost averaging analysis

### Data Acquisition Strategy
- **Phase 1 (MVP)**: Manual CSV/HTML file import (user downloads from securities site → uploads to app)
- **Phase 2+ (Future)**: Semi-automated web scraping with user-assisted authentication (Puppeteer + manual 2FA input)
- **Limitation**: Full automation blocked by passkey/2FA requirements on both Rakuten and SBI
- No official APIs available from Rakuten Securities or SBI Securities

### Exchange Rate & Currency
- Daily USD/JPY rate from Yahoo Finance API or Open Exchange Rates
- Cache most recent rate for offline use
- Display conversion rate and timestamp on UI

### Multi-Currency Support
- Japanese domestic assets: JPY only
- US assets (ETFs, stocks): Display in both USD and JPY
- Exchange rate source: TBD in design phase

### Dividend and Distribution Tracking
- Store payment history with date, amount, currency
- Calculate total return = capital gains + cumulative income
- Monthly aggregation for trend analysis
- Support both stock dividends and fund distributions

### Monthly Reporting
- Automated monthly summary generation
- Include: asset value change, dividend/distribution income, performance metrics
- Aligned with monthly dollar-cost averaging investment cadence

### Data Protection & Logging
- **Encryption**: Master key (256-bit) + user password (PBKDF2/Argon2)
- **Backup**: Weekly auto-backup to local JSON file (timestamped)
- **Restore**: Merge or replace options on import
- **Logging**: 30-day retention, excludes credentials

### UI/UX Specifications
- **Charting**: Interactive library (Chart.js/Recharts) with zoom, pan, time range selection
- **Screen Size**: Desktop/tablet (1024px+) primary target
- **Mobile**: Not supported in Phase 1 (deferred)
